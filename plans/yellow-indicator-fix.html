Yellow Indicator Fix

TerminalPane.tsx · Sidebar.tsx — Agent running-state detection

Indicator Legend

Yellow blinking — Agent is actively working (tool calls, thinking, generating output)
Green solid — Agent session exists but is idle at its prompt
Red solid — Session stale (no activity for 5+ minutes)

How It Works Today

State Machine — AgentState

Defined in App.tsx: 'none' | 'running' | 'idle'. Lives on each Session object. Flows: TerminalPane → onAgentStateChange prop → App → Sidebar → dot CSS class.

The dot is yellow/blinking only when session.agentState === 'running' (getSessionDotStatus in Sidebar.tsx).

State Transitions in TerminalPane.tsx
FunctionSets state toWhen called
markAgentRunning()'running'User presses Enter with an agent command or in an agent session; PTY data arrives while agentTaskInFlightRef.current is true
markAgentIdle()'idle'Idle timer fires after AGENT_IDLE_DELAY_MS (6 s) of no PTY output
clearAgentSession()'none'PTY exits; shell process detected and agent session is gone

Root Causes

CriticalBug 1 — agentTaskInFlightRef gate breaks re-arm after idle timer fires

In the PTY output handler (onPtyData, line 823), the indicator only re-arms if agentTaskInFlightRef.current is true:

// TerminalPane.tsx ~line 823
const removeDataListener = window.api.onPtyData(session.id, (data) => {
  touchSessionActivity()
  if (agentTaskInFlightRef.current) {       // <-- the gate
    markAgentRunning(agentProcessRef.current)
  }
  ...

markAgentIdle() sets agentTaskInFlightRef.current = false. So once the 6-second timer fires — even mid-task — subsequent PTY output never re-triggers 'running'. The indicator goes dark and stays dark for the rest of the task.

CompoundingBug 2 — Idle delay (6 s) is shorter than typical Claude API round-trips

Claude API calls with thinking enabled, large context, or slow tools regularly take 10–45 seconds to return. The constant AGENT_IDLE_DELAY_MS = 6_000 causes the idle timer to fire routinely during normal task execution, not just after a task completes.

Failure Scenario Walkthrough

User types task + Enter running
Enter key → agentSessionRef.current is true → markAgentRunning() → agentTaskInFlightRef = true, idle timer = 6 s. Indicator blinks yellow. ✓
Claude generates initial output (0–3 s) running
PTY data flows → agentTaskInFlightRef.current is true → markAgentRunning() resets idle timer. ✓
Claude calls a tool (e.g. Bash, Read) — PTY goes quiet running
No PTY output while Claude waits for tool execution or the Anthropic API. Idle timer counting down. ✓ (still correct)
6-second idle timer fires goes idle ✗
markAgentIdle() → agentTaskInFlightRef = false, state = 'idle'. Indicator turns green even though Claude is still mid-task.
Tool finishes — Claude resumes, PTY output arrives stays idle ✗
PTY data fires but agentTaskInFlightRef.current is now false → gate blocks → markAgentRunning() is never called. Indicator stays green for the rest of the task, through all further tool calls and output.
Task completes, Claude returns to prompt idle
State was already (wrongly) idle, so nothing changes.

Proposed Fix

Two-line change in TerminalPane.tsx

No new state, no new refs. The existing agentSessionRef already correctly tracks whether an agent process is running in this terminal.

Change 1 — Increase idle delay (line 13)

src/renderer/src/components/TerminalPane.tsx · line 13
const AGENT_IDLE_DELAY_MS = 6_000
const AGENT_IDLE_DELAY_MS = 20_000

20 seconds covers the vast majority of Claude API round-trips (including thinking mode). It means the indicator may stay yellow up to 20 s after a task completes — acceptable latency vs. the current bug of it going dark mid-task.

Change 2 — Fix the PTY data re-arm gate (lines 823–825)

src/renderer/src/components/TerminalPane.tsx · onPtyData handler, ~line 823
const removeDataListener = window.api.onPtyData(session.id, (data) => {
  touchSessionActivity()
  if (agentTaskInFlightRef.current) {
  if (agentSessionRef.current) {
    markAgentRunning(agentProcessRef.current)
  }
  ...

Replace the agentTaskInFlightRef gate with agentSessionRef.current. Now any PTY output while an agent is in session re-arms the running state and resets the idle timer. This directly fixes the scenario where the idle timer fired mid-task: the moment Claude resumes printing output, the indicator snaps back to yellow.

Why this is safe: Claude Code in alt-screen (TUI mode) only generates PTY output when it's actually doing something — rendering output, tool results, thinking blocks. It does not emit a continuous stream of bytes at its idle prompt. The 20-second timer handles the transition back to idle after Claude returns to its prompt.

What happens to agentTaskInFlightRef?

After this change, agentTaskInFlightRef is still set to true by markAgentRunning() and false by markAgentIdle() and clearAgentSession() — its writes are unchanged. Its reads are just no longer the gate for PTY re-arm. It can be removed in a follow-up cleanup, but touching it here is unnecessary scope.
