Tab Ordering Fix

Plan · emu · May 2026

Problem

Tabs in the left sidebar reorder whenever any session activity occurs — including Claude Code output arriving in a background tab. The user only wants the order to change when they physically click (or navigate to) a tab.

Root cause: the Session object has a single lastActiveAt timestamp that is updated both for user navigation and for background PTY activity. The sidebar sorts by this field, so any background activity reshuffles the list mid-session.

Primary Symptom: Tab Flickering

⚡ Rapid tab flashing with two concurrent Claude Code sessions

When two tabs both have Claude Code actively running (yellow indicator blinking), each stream of PTY output calls touchSession, bumping that tab's lastActiveAt to right now. Since both sessions are streaming output simultaneously, they alternate being the most-recently-active — whichever received a byte last jumps to position #1. This causes the two tabs to swap positions dozens of times per second, producing visible flicker in the sidebar.

The fix eliminates this entirely: PTY activity will never touch the sort key. Tabs only move when the user explicitly navigates to one.

Root Cause — Call Map

Every call to touchSession(id) bumps lastActiveAt, which the sidebar sorts by.

Call site in App.tsx Triggered by Should reorder?
handleSelect User clicks a tab Yes
handleNewSession User clicks "+" (new tab) Yes
handleDropLeft / handleDropRight User drags a tab into a pane Yes
handleCloseLeftPane User closes a pane; right session is promoted Yes
handleOpenHistory User clicks history icon on a tab No — keep current order
onSessionTouched → touchSessionActivity (TerminalPane) PTY data arrives — Claude Code / any background process No — this is the bug

Solution

Introduce a second timestamp, userSelectedAt, on the Session type. It is initialised to createdAt and only ever updated when the user explicitly navigates to a tab. The sidebar sorts by userSelectedAt; lastActiveAt is kept purely for the relative-time display label ("just now", "5 min ago").

src/renderer/src/App.tsx

Add a selectSession helper that bumps userSelectedAt

Keep touchSession exactly as-is (it still drives the timestamp label). Add a tiny helper that also bumps the sort key:

const selectSession = useCallback((id: string) => {
  const now = new Date()
  setSessions((prev) =>
    prev.map((s) => s.id === id ? { ...s, lastActiveAt: now, userSelectedAt: now } : s)
  )
}, [])

One-line change in orderedSessions:

The fallback sort (createdAt) is unchanged — it handles ties between tabs the user has never manually visited.

End Result

No more tab flickering

Running two or more concurrent Claude Code sessions will no longer cause tabs to flash or swap positions. The sidebar stays completely still while background tasks run — regardless of how much output they produce or how fast they stream it.

Tab order reflects your navigation, not machine activity

The sidebar becomes a reliable history of where you have been. The tab you most recently clicked sits at the top. Tabs you haven't visited stay where you left them, even if Claude Code has been streaming output in them for minutes.

Relative timestamps still live

The "just now / 5 min ago" label under each tab name continues to update in real time based on PTY activity — so you can still glance at the sidebar and know which sessions have been recently active, without that activity disturbing the order.

New tabs still appear at the top

Creating a new session initialises userSelectedAt to the current time, so it naturally lands at position #1 — consistent with the existing expectation that a freshly opened tab is the one you want to use.

Zero regressions to existing UX

Drag-to-pane, close-pane promotion, split-screen, and sidebar collapse all behave identically to today. The only observable change is that background agent activity no longer has any effect on sidebar order.

Scope & Risk

Minimal blast radius

No IPC changes, no new components, no new dependencies. The diff is ~15 lines across two files. lastActiveAt continues to work exactly as before for timestamp display; only the sort key is swapped.
