Native Markdown Viewer Implementation Plan

Emu feature plan - click a Markdown path in terminal output and render it in a collapsible right-side pop-out.

Goal: When a user Cmd-clicks a local Markdown file link in Emu, Emu should open a native, read-only Markdown viewer in a right-hand pop-out pane instead of sending the file to Finder or the system default app.

Core assumption: "Markdown file link" means a local path visible in terminal output, including absolute paths, relative paths, and file:// links that point to .md-style files. Web URLs should continue opening in the browser.

Current Codebase Facts

Area What exists today Implication
TerminalPane.tsx Registers xterm link providers for web URLs, bare domains, and local-looking file paths. The click interception point already exists. We should extend it rather than adding a separate global click layer.
preload/index.ts Exposes safe IPC methods such as openExternal and openPath. Markdown file reads must be added here because the renderer is sandboxed and has no Node filesystem access.
main/index.ts Validates external URLs and absolute paths before delegating to Electron shell APIs. File validation belongs in the main process. The renderer should never decide that a local file is safe to read.
App.tsx Owns terminal sessions, split panes, active pane state, and settings overlays. The Markdown pop-out should be owned here so it can sit on the right edge of the app without becoming a terminal session or changing split-screen state.
package.json No Markdown renderer dependency is installed. Add a small renderer dependency such as markdown-it, configured with raw HTML disabled.

Product Shape

Open behavior Cmd-click README.md, ./docs/foo.md, /abs/file.md, or file:///abs/file.md opens the Markdown in a right-hand pop-out.
Viewer surface A right-side pop-out overlays the terminal area, keeps terminals alive, and has Preview / Markdown, Collapse, Close, filename, refresh, and open externally controls.
Markdown links Links inside rendered Markdown keep web URLs external, open local Markdown internally, and use the existing path opener for other local files.
View modes Preview mode renders rich text by default. Markdown mode shows the raw source in a monospaced, selectable, scrollable code view.
The most important technical gap is current working directory tracking. Existing relative path detection advertises ./relative links, but shell:openPath only accepts absolute paths. Native Markdown support should fix that properly rather than guessing from the app launch directory.

Pop-Out Behavior

The Markdown viewer should behave like a right-hand document drawer, not like a terminal tab. Opening Markdown should not create a new PTY session, rename a tab, or force split-screen mode.

  • Expanded: The pop-out is anchored to the right edge of .terminal-area, overlays the terminal surface, and uses a fixed responsive width such as min(520px, 42vw).
  • Collapsed: Pressing Collapse shrinks it to a narrow right-edge rail that shows a document icon, filename tooltip, and expand button. Terminal interaction should be available across the remaining space.
  • Closed: Pressing Close removes the Markdown document from view entirely. Re-clicking any Markdown link opens the latest document expanded again.
  • Split-screen: The pop-out still anchors to the far right of the app. It overlays whichever terminal content is on the right side without reassigning the right split pane.
  • Focus: Opening the pop-out moves keyboard focus into the document. Closing or collapsing returns focus to the active terminal pane.

Preview And Markdown Modes

The pop-out should match standard Markdown file viewers: users can toggle between a rendered Preview mode and a raw Markdown source mode.

  • Preview mode: Default mode. Render headings, paragraphs, lists, tables, links, code blocks, blockquotes, and images using markdown-it with raw HTML disabled.
  • Markdown mode: Show the exact file text from disk in a read-only monospaced source view. Preserve whitespace, indentation, fenced code blocks, and line wrapping. The source should be selectable and copyable.
  • Toggle control: Add a segmented control in the toolbar labeled Preview and Markdown. The active segment should be visually clear and keyboard reachable.
  • Scroll behavior: Preserve independent scroll positions for Preview and Markdown modes while the same document is open.
  • Collapse behavior: Collapsing and expanding the pop-out should preserve the selected mode.

Architecture

Terminal output
  -> xterm link provider in TerminalPane
  -> resolve clicked path using pane cwd
  -> main-process validation and Markdown read IPC
  -> App-level Markdown pop-out opens on the right side
  -> MarkdownPopout toggles between preview render and raw source
  -> internal link clicks recurse through the same resolver

Implementation Phases

1

Foundation Add current-directory tracking

Add a small Emu-specific shell integration sequence that announces the session cwd before each prompt. Store it in TerminalPane as currentWorkingDirectoryRef.

  • In setupShellIntegration(), append a zsh precmd hook that emits a private OSC sequence such as ESC ] 633;EmuCwd=/absolute/path BEL.
  • In TerminalPane.tsx, parse incoming PTY data for that OSC marker before terminal.write(data).
  • Default cwd to the PTY spawn directory for sessions that have not emitted the marker yet.
  • Do not expose cwd in UI; it is only used to resolve relative links.
type PaneCwdState = {
  cwd: string
  updatedAt: number
}
2

Safety Add main-process Markdown file IPC

Create validation helpers in main/index.ts so the renderer can ask for Markdown content without direct filesystem access.

  • Accept only absolute resolved file paths after combining raw link text with the pane cwd.
  • Allow extensions .md, .markdown, .mdown, and .mkd.
  • Reject null bytes, directories, missing files, huge files, and non-UTF-8 read failures with user-readable errors.
  • Return metadata: normalized path, basename, parent directory, file size, mtime, and Markdown text.
interface MarkdownOpenResult {
  ok: true
  path: string
  name: string
  directory: string
  markdown: string
  size: number
  mtimeMs: number
}
3

Routing Change file link activation

Replace the current activate: () => window.api.openPath(path) behavior with a resolver that branches by file type and notifies App.tsx when Markdown should open.

  • If the clicked path resolves to Markdown, call the new Markdown IPC and invoke onOpenMarkdown(result).
  • If the clicked path is not Markdown, keep using openPath.
  • Add support for file:// Markdown URLs in a custom link provider because safe external URLs intentionally exclude local file URLs.
  • Keep web URLs and bare domains unchanged.
async function openClickedFile(rawPath: string) {
  const result = await window.api.markdownOpen({ rawPath, cwd: currentWorkingDirectoryRef.current })
  if (result.ok) {
    onOpenMarkdown(result)
    return
  }
  if (result.reason === 'not-markdown') {
    await window.api.openPath(result.path)
  }
}
4

Viewer Build the right-hand Markdown pop-out

Add MarkdownPopout.tsx and render it from App.tsx as a right-edge overlay inside .terminal-area. Terminal components stay mounted, preserving PTY output, scrollback, agent state, and split-pane behavior.

  • Toolbar: Preview / Markdown segmented control, Collapse, Close, filename, truncated full path, refresh, and open externally.
  • Expanded layout: right-anchored drawer with min(520px, 42vw) width and full height below the titlebar.
  • Collapsed layout: right-edge rail with an expand button and filename tooltip; document state remains loaded.
  • Preview content: Markdown rendered with markdown-it, html: false, and link handling through React click interception.
  • Markdown content: Raw file text rendered in a read-only <pre> / <code> source view with monospaced font, selectable text, and preserved whitespace.
  • Mode state: Store markdownViewMode as 'preview' | 'source'. Default to 'preview' whenever a new document opens, and preserve the mode across collapse/expand for the current document.
  • Scroll state: Keep separate preview and source scroll offsets so toggling does not feel like losing place.
  • States: loading, loaded, failed read, unsupported file, file changed on disk.
  • Keyboard: Escape collapses when expanded; Escape closes when already collapsed; focus returns to the active terminal pane.
<MarkdownPopout
  document={markdownDocument}
  collapsed={markdownCollapsed}
  viewMode={markdownViewMode}
  onViewModeChange={setMarkdownViewMode}
  onCollapse={() => setMarkdownCollapsed(true)}
  onExpand={() => setMarkdownCollapsed(false)}
  onClose={() => setMarkdownDocument(null)}
  onRefresh={() => reopenMarkdown(markdownDocument.path)}
  onOpenLink={(href) => openMarkdownRelativeLink(href, markdownDocument.directory)}
/>
5

Links Make rendered Markdown links feel native

Intercept clicks inside the rendered Markdown document instead of allowing Chromium navigation.

  • https:// and http:// links call openExternal.
  • Relative Markdown links resolve against the current document directory and replace the viewer content.
  • Relative non-Markdown file links resolve and call openPath.
  • Anchor-only links such as #usage scroll within the viewer.
  • Link interception only applies in Preview mode. Markdown mode displays raw source as text, so Markdown syntax should not become clickable there.
6

Assets Support local images after text rendering works

For seamless Markdown, local images should render instead of showing broken file:// references.

  • Add a main-process helper that reads allowed local image extensions and returns a data URL with a size cap.
  • Rewrite relative Markdown image URLs before rendering or post-process img tags after render.
  • Allow only common image extensions: .png, .jpg, .jpeg, .gif, .webp, .svg if SVG sanitization is explicitly handled; otherwise defer SVG.
7

Polish Update docs and manual QA coverage

Once behavior is implemented, update the user-facing feature list and run a focused verification pass.

  • Update README.md: Markdown paths open in Emu; other file paths still open through the system.
  • Run npm run build to verify TypeScript and bundling.
  • Run the app and manually test absolute paths, relative paths after cd, split panes, file:// Markdown URLs, broken files, large files, and links inside rendered Markdown.

Security Rules

  • Keep nodeIntegration: false, contextIsolation: true, and renderer sandboxing unchanged.
  • Never let rendered Markdown execute raw HTML or scripts. Configure the Markdown renderer with raw HTML disabled.
  • Never navigate the Electron window to clicked Markdown links. Route all link clicks through explicit IPC helpers.
  • Read files only through main-process IPC with extension, type, size, and path validation.
  • Keep openExternal limited to http: and https:.

Recommended First Cut

Build phases 1 through 5 first. That delivers the central user promise: local Markdown links open in a right-hand Emu pop-out, Preview mode renders rich Markdown, Markdown mode shows raw source, relative links work, Markdown-to-Markdown navigation stays native, and the user can collapse the document when they need more terminal space. Then add local images in phase 6 before calling the feature polished.

This keeps the first implementation surgical: one new pop-out component, one CSS file, focused IPC additions, app-level viewer and mode state, and a targeted change to the existing terminal link provider.
