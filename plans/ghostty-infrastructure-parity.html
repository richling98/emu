Ghostty Infrastructure Parity

Competitive analysis & improvement roadmap · emu · May 2026

Executive Summary

Ghostty is the most technically ambitious terminal emulator on the market — 54k GitHub stars, millions of daily users, written in Zig + Swift with a custom Metal renderer, a SIMD-accelerated VT parser, and a native multi-threaded architecture that matches Alacritty on benchmarks while being far more feature-rich.

Emu is built on Electron + xterm.js. This gives us an outstanding UI/UX foundation and rapid iteration speed, but creates a fundamental performance gap in raw throughput, latency, and protocol coverage. This plan identifies what we can close within the Electron model and where we ultimately need to go further.

Emu's differentiators — AI prompt optimizer, agent status detection, command history with smart output capture — are not in Ghostty at all. We shouldn't abandon them. The goal is to close the infrastructure gap while keeping those product advantages.

The Real Answer: A Native Swift Rebuild

True infrastructure parity with Ghostty requires leaving Electron. Phases 1–3 below are genuine improvements worth shipping — but they have a ceiling. No combination of native addons, SharedArrayBuffers, or xterm.js tuning can match a Metal renderer + CoreText + a multi-threaded native binary. This section explains what true parity actually looks like and why it's achievable without writing Zig from scratch.

The Honest Truth About the Architecture Gap

Electron has a ceiling. No amount of tuning will make Chromium's WebGL pipeline match Ghostty's custom Metal renderer. The IPC hop between main and renderer adds inherent latency. xterm.js is excellent JavaScript, but it isn't SIMD-accelerated Zig. We should close every gap we can within the Electron model — but we should be clear-eyed that some gaps require a deeper architectural change to close fully.
The plan below is phased by effort and impact. Phases 1–2 can be done now, within the existing codebase. Phase 3 requires a native addon. Phase 4 is the strategic rearchitecture question we'll need to answer if we want true parity.

Lock in the WebGL renderer — never fall back to canvas

xterm.js silently falls back to a Canvas2D renderer when WebGL context creation fails (low memory, too many canvases, headless mode). Canvas2D is ~10× slower. We should detect the fallback and report it, and force WebGL options that make a fallback less likely. 

// Add after addon load:
const webglAddon = new WebglAddon()
webglAddon.onContextLoss(() => {
  // reload the addon — this keeps WebGL alive after GPU context loss
  webglAddon.dispose()
  terminal.loadAddon(new WebglAddon())
})
terminal.loadAddon(webglAddon)

Batch PTY output writes — eliminate per-byte IPC round-trips

Right now every data event from node-pty fires a separate pty:data IPC event to the renderer. High-throughput programs (e.g. cat large_file) can produce thousands of events per second, hammering the IPC bridge. We should coalesce writes in a short timer window (≤4 ms — one animation frame). 

// In pty:create handler, replace direct emit with a batching queue:
const buffers = new Map<string, string[]>()
const flushTimers = new Map<string, NodeJS.Timeout>()

ptyProcess.onData((data) => {
    event.sender.send('pty:data', { id, data })
    if (!buffers.has(id)) buffers.set(id, [])
    buffers.get(id)!.push(data)
    if (!flushTimers.has(id)) {
      flushTimers.set(id, setTimeout(() => {
        const chunk = buffers.get(id)!.join('')
        buffers.delete(id)
        flushTimers.delete(id)
        event.sender.send('pty:data', { id, data: chunk })
      }, 4))
    }
})
Impact
High

OSC 133 shell integration — replace the ZDOTDIR zsh hack

Emu currently injects prompt-detection via a ZDOTDIR wrapper that patches zsh startup. This is fragile (zsh only, breaks if users have custom ZDOTDIR, can conflict with frameworks like Oh My Zsh). Ghostty uses the OSC 133 shell integration protocol — the same protocol used by iTerm2, WezTerm, and kitty. It works via a few escape sequences that any shell can emit via ~/.zshrc snippets, and it's shell-agnostic.

xterm.js already parses OSC sequences. We need to:

StepDescription
1. Write shell snippetsSmall ~/.config/emu/shell-integration.zsh and .bash files that emit OSC 133 A/B/C/D marks at prompt and command boundaries
2. Source them from emuSet EMU_SHELL_INTEGRATION=1 in the spawned PTY env; snippets check this var before activating
3. Parse in rendererHandle OSC 133 sequences in TerminalPane.tsx to mark prompt lines — replace the current buffer-scan logic
4. Remove ZDOTDIR wrapperDelete zdotdir/ and related main-process setup
Impact
High (reliability + bash/fish support)

src/renderer/src/components/TerminalPane.tsx

Synchronized output rendering (DEC mode 2026)

Ghostty supports synchronized rendering — an escape sequence (ESC[?2026h) that lets programs (like vim, tmux, neovim) tell the terminal "batch this output, don't render until I say go." This eliminates screen tearing and partial-render flicker in TUI apps. xterm.js v5 supports it natively; we just need to enable it. 

const terminal = new Terminal({
    allowProposedApi: true,   // required for sync render
    // rest of options...
})

Synchronized output is already in xterm.js 5.x. Enabling allowProposedApi unlocks it with zero additional code.

Impact
Medium (visible quality in TUI apps)

src/renderer/src/components/TerminalPane.tsx

Kitty keyboard protocol

Ghostty supports the Kitty keyboard protocol, which allows applications like Neovim and Helix to receive key events with full modifier information that traditional VT100 sequences can't express (e.g., Ctrl+Shift+Enter, Ctrl+i vs Tab). This is a significant quality-of-life feature for power users of modern terminal editors.

xterm.js 5.x has a customKeyEventHandler API. A Kitty protocol addon is also in active development in the xterm.js community. We should track and integrate the first stable implementation.

Impact
Medium (critical for Neovim/Helix users)

SharedArrayBuffer PTY bridge — eliminate IPC serialization

The biggest remaining bottleneck in the Electron model is the IPC bridge between the main process (where PTY lives) and the renderer (where xterm.js lives). Every byte of terminal output crosses this bridge as a serialized string IPC message. Electron supports SharedArrayBuffer (SAB), which lets the main process write PTY bytes directly into memory the renderer can read without serialization.

ApproachMechanismLatency reduction
CurrentipcMain.emit → structured clone → ipcRenderer.onbaseline
Phase 1 batchSame, but coalesced over 4ms~50% on throughput tasks
SAB ring bufferMain writes to SAB ring; renderer polls via Atomics.waitAsync~70–80% on latency

Implementation requires enabling crossOriginIsolation in the Electron BrowserWindow (sets COOP/COEP headers). This is a breaking change to the renderer's security context but is well-supported in modern Electron.

High (biggest latency win in Electron model)

Native Node.js addon for VT state — embed libghostty-vt

Ghostty has released libghostty-vt as a standalone C/Zig library for VT sequence parsing and terminal state management. It's available for macOS, Linux, Windows, and WebAssembly. We can use it as a Node.js native addon (N-API) to replace xterm.js's JavaScript VT parser for the hot path.

The architecture would be:

This is the highest-leverage single change we can make — the VT parser and state machine are the bottleneck for high-throughput output (e.g., find / or large log files).

This is not optional if infrastructure parity is the actual goal. Phases 1–3 are genuine improvements and worth shipping now. But they are incremental gains within a fundamentally limited architecture. Phase 4 is where emu becomes a peer to Ghostty, not just a well-tuned Electron app.
The Native Path: Swift + libghostty-vt + Metal

Rebuild emu as a native macOS app using SwiftUI for the shell, libghostty-vt (Ghostty's open-source C/Zig library) for VT parsing, and a custom Metal renderer with CoreText for fonts. This is exactly how Ghostty is built — and since libghostty-vt is open source, we inherit Ghostty's SIMD-accelerated parser without writing any Zig ourselves.

All of emu's AI features (prompt optimizer, agent detection, command history) port cleanly to Swift. URLSession replaces fetch. State machines translate directly. The terminal domain knowledge we've built up stays intact.

Intermediate option: Tauri (Rust + system WebView)

Tauri replaces Electron's Chromium with WKWebView (macOS system WebView), cutting memory by ~70% and removing Chromium overhead. The React + xterm.js renderer stays intact. This is a 2–4 week migration and gets us partway there — better memory and startup, but still no Metal renderer or CoreText. A useful stepping stone if the full native rebuild feels too large to commit to right now.

Where Emu Already Wins vs Ghostty

These are our moats — protect and extend them

AI Prompt Optimizer — No other terminal emulator has this. Ghostty has explicitly stated they have no AI features. This is a differentiated product advantage.

Agent Status Detection — Knowing when Claude Code is running/idle/stale, and surfacing that in the sidebar, is unique to Emu. Ghostty just shows processes.

Command History with Smart Output Capture — Ghostty has no equivalent. Emu's ability to browse past commands with searchable output previews is a genuine productivity feature that power users notice immediately.

UI Polish and Themes — Emu's design aesthetic (Catppuccin-first, dark and beautiful) is subjectively better than Ghostty's more utilitarian look. Continue investing here.

Recommendation

The two-track strategy: ship Phase 1–2 improvements now (days of effort, immediate user-visible gains), while planning the native rebuild in parallel. Don't wait for Phase 3 — the SharedArrayBuffer work and libghostty-vt native addon are valuable engineering experience that directly informs the native rebuild, but they aren't prerequisites for it.
Now: Phase 1 quick wins — ~3 days

WebGL lock, PTY batching, xterm.js tuning, and enabling the Electron sandbox. Zero architectural risk. Ship as a "performance & reliability" release. These buy goodwill with users while the native rebuild is planned.

Next: Phase 2 protocol parity — ~1 week

OSC 133 shell integration (removes the zsh-only ZDOTDIR hack) and synchronized rendering. These are the most user-visible gaps vs Ghostty and close real complaints from power users who use bash, fish, or neovim.

Strategic: Begin native Swift rebuild planning — 10–14 weeks to v1

This is the real answer to "parity with Ghostty." Start with a Swift prototype: PTY management + libghostty-vt bindings + a minimal Metal renderer. Once text renders correctly in a native window, the rest (SwiftUI shell, AI features, themes, settings) follows quickly. The Electron app stays live during the rebuild — users see no disruption.

Ghostty's macOS app source code is open. We can study exactly how they structure their SwiftUI app, their Metal renderer, and their libghostty-vt integration. We're not starting from zero — we're starting from a proven blueprint.

Skip if not ready: Tauri as an intermediate step

If committing to the full native rebuild feels too large right now, Tauri is a valid 2–4 week intermediate step that cuts memory and removes Chromium. It doesn't achieve infrastructure parity, but it buys time and reduces Electron-specific complaints while the native app is built in parallel.
